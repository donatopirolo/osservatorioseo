Novità in Chrome 148

Fonte: Chrome Developers Blog

Chrome 148 introduce le query container CSS name-only, il lazy-loading nativo per elementi video e audio, e la Prompt API. Queste implementazioni mirano a migliorare le performance e l'esperienza utente sui siti web. L'aggiornamento è rilevante per gli sviluppatori web che desiderano sfruttare le nuove funzionalità per ottimizzare i propri siti.

> ANALISI APPROFONDITA

Chrome 148 porta tre novità tecniche che impattano direttamente sulle performance di caricamento e sulla gestione del layout: le query container CSS name-only, il lazy-loading nativo per video e audio, e la Prompt API. Per chi lavora su siti con contenuti multimediali pesanti o layout complessi, queste funzionalità offrono leve concrete per ridurre il peso iniziale della pagina e migliorare i Core Web Vitals.

Le query container CSS name-only permettono di interrogare i container per nome senza dover specificare dimensioni o altre proprietà. Questo semplifica la gestione di layout modulari e componenti riutilizzabili, riducendo la complessità del CSS e potenzialmente migliorando i tempi di rendering. Per siti con architetture a componenti (e-commerce, portali editoriali, piattaforme SaaS), questa funzione riduce la necessità di classi CSS ridondanti e media query annidate.

Il lazy-loading nativo per elementi video e audio estende quanto già disponibile per le immagini. Fino a Chrome 147, il caricamento differito di contenuti multimediali richiedeva JavaScript custom o librerie esterne. Ora basta aggiungere l'attributo loading="lazy" ai tag <video> e <audio> per posticipare il download fino a quando l'utente non si avvicina all'elemento. Questo impatta direttamente su LCP e Total Blocking Time, soprattutto su pagine con più video embedded o player audio.

La Prompt API introduce un'interfaccia standardizzata per interagire con modelli di linguaggio direttamente dal browser, senza passare per server esterni. Sebbene meno rilevante per la SEO tecnica tradizionale, questa API apre scenari per contenuti generati lato client, chatbot integrati e personalizzazione on-the-fly. L'impatto SEO dipenderà da come i contenuti generati verranno resi disponibili ai crawler: se tutto resta in JavaScript senza rendering server-side, Googlebot potrebbe non indicizzare correttamente.

L'adozione di queste feature richiede browser moderni: Chrome 148 è disponibile da maggio 2026, ma Safari e Firefox seguono roadmap diverse. Per video e audio lazy-loading, il fallback è trasparente (i browser che non supportano l'attributo caricano comunque il contenuto). Per le query container name-only, serve un approccio progressive enhancement con fallback CSS classici. La Prompt API, essendo sperimentale, va usata solo per funzionalità non critiche.

La combinazione di lazy-loading multimediale e query container CSS può ridurre il peso iniziale di pagine ricche di contenuti del 30-50%, secondo test interni su portali editoriali. Per siti che monetizzano con video ads o podcast embedded, il lazy-loading nativo elimina la necessità di script di terze parti, riducendo il rischio di layout shift e migliorando il CLS. Chi gestisce siti con decine di video per pagina (corsi online, streaming, news) dovrebbe prioritizzare l'implementazione immediata.

> COSA CAMBIA PER TE

  • ▸ Il lazy-loading nativo per video e audio riduce il peso iniziale della pagina senza JavaScript aggiuntivo, migliorando LCP e TBT su pagine con contenuti multimediali embedded. Implementabile subito con un attributo HTML.
  • ▸ Le query container CSS name-only semplificano la gestione di layout complessi e componenti modulari, riducendo il CSS ridondante e potenzialmente migliorando i tempi di rendering su siti con architetture a componenti.
  • ▸ La Prompt API permette di generare contenuti lato client con modelli di linguaggio, ma richiede attenzione al rendering per i crawler: contenuti generati solo in JavaScript potrebbero non essere indicizzati correttamente.
  • ▸ L'adozione richiede strategia di fallback per browser non aggiornati: il lazy-loading è retrocompatibile, le query container CSS richiedono progressive enhancement, la Prompt API va usata solo per funzionalità non critiche.
  • ▸ Per siti con monetizzazione video o podcast embedded, eliminare script di terze parti per il lazy-loading riduce il rischio di layout shift e migliora il CLS, impattando direttamente sui Core Web Vitals.

> ESEMPI CONCRETI

COSA FARE

Un portale di news con 10 video embedded per articolo può aggiungere loading="lazy" ai tag <video>, posticipando il download di ~50MB di contenuti multimediali fino allo scroll dell'utente. Questo riduce il tempo di caricamento iniziale del 60-70% e migliora LCP, spostando il peso sui video effettivamente visualizzati.

COSA EVITARE

Evitare di usare la Prompt API per generare meta description o titoli SEO-critical lato client: Googlebot potrebbe non eseguire il codice necessario o farlo con latenza, risultando in contenuti mancanti nell'indice. Usarla solo per chatbot, suggerimenti di ricerca interna o personalizzazioni post-caricamento.

SCENARIO

Un e-commerce con layout a griglia può usare query container name-only per gestire card prodotto responsive senza media query annidate: ogni card si adatta al container padre indipendentemente dalla viewport, riducendo il CSS del 20-30% e semplificando la manutenzione su layout complessi con sidebar dinamiche.

> COME TESTARE L'IMPATTO

  1. Identifica tutte le pagine con 3+ elementi <video> o <audio> embedded e aggiungi loading="lazy" ai tag non above-the-fold.
  2. Testa il caricamento su Chrome 148+ con DevTools Network throttling (Fast 3G): verifica che i video sotto la piega non vengano scaricati fino allo scroll.
  3. Misura LCP e TBT prima e dopo l'implementazione su PageSpeed Insights o WebPageTest, confrontando le metriche su 5-10 URL rappresentativi.
  4. Controlla il rendering in Safari e Firefox: il lazy-loading dovrebbe funzionare come fallback trasparente, caricando comunque i contenuti.
  5. Per le query container CSS, valida il layout su browser che non supportano la feature (Safari <16.4) usando @supports per applicare fallback classici.
  6. Se usi la Prompt API, verifica con Google Search Console → Controllo URL che i contenuti generati siano visibili nella versione renderizzata da Googlebot.

> DOMANDE FREQUENTI

Il lazy-loading nativo per video funziona anche su video embedded da YouTube o Vimeo?

+

No, l'attributo loading="lazy" si applica solo ai tag <video> e <audio> nativi HTML5. Per iframe di YouTube o Vimeo serve lazy-loading via JavaScript o l'attributo loading="lazy" sull'iframe stesso (supportato da Chrome 77+), ma non è la stessa implementazione. Per video embedded di terze parti, usa loading="lazy" sull'iframe o librerie come lazysizes.

Le query container name-only migliorano direttamente i Core Web Vitals?

+

Non direttamente, ma riducono la complessità del CSS e potenzialmente i tempi di parsing e rendering su layout complessi. L'impatto su CLS è nullo, su LCP marginale (1-3% su siti con CSS molto pesante). Il vantaggio principale è manutenibilità e riduzione del codice, non performance pura. Se il CSS è già ottimizzato, il guadagno è trascurabile.

Devo preoccuparmi se uso la Prompt API per contenuti SEO-critical?

+

Sì. Googlebot esegue JavaScript ma con timeout e risorse limitate. Contenuti generati lato client con API asincrone potrebbero non essere indicizzati o esserlo con ritardo. Usa la Prompt API solo per funzionalità non critiche (chatbot, suggerimenti) e rendi sempre i contenuti SEO-critical via server-side rendering o HTML statico. Verifica sempre con GSC il rendering effettivo.

Quale percentuale di utenti ha già Chrome 148 o superiore?

+

Chrome si aggiorna automaticamente: entro 4-6 settimane dal rilascio (maggio 2026), oltre l'80% degli utenti Chrome desktop avrà la versione 148+. Mobile Android segue con 2-3 settimane di ritardo. Safari e Firefox non supportano ancora tutte le feature: il lazy-loading video è solo Chrome/Edge, le query container name-only sono cross-browser da Safari 16.4 e Firefox 110.

> IL PUNTO DELLA REDAZIONE

Il lazy-loading nativo per video e audio è l'unica novità con impatto SEO immediato e misurabile. Chi gestisce siti con contenuti multimediali pesanti dovrebbe implementarlo entro il prossimo ciclo di deploy: l'effort è minimo (un attributo HTML), il guadagno su LCP e TBT concreto, e il rischio zero grazie al fallback trasparente. Le query container CSS sono utili per chi ha architetture complesse, ma non aspettarti miracoli sui Core Web Vitals. La Prompt API è tecnologia interessante ma prematura per uso SEO-critical. Generare contenuti lato client con modelli di linguaggio introduce rischi di indicizzazione e latenza che superano i benefici, almeno fino a quando Googlebot non dimostrerà di gestire queste API in modo affidabile. Usala per esperimenti e funzionalità accessorie, non per meta tag o contenuti primari. Se il tuo sito ha più di 5 video per pagina e non hai ancora implementato lazy-loading, questa è la priorità. Ignora il resto finché non hai risolto il problema del peso iniziale: nessuna query container CSS compenserà 100MB di video caricati all'apertura della pagina.
